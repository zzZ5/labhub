# LabHub 优化推进清单

本文档用于后续持续整理 LabHub。按阶段顺序推进：当前阶段通过验收后，再进入下一阶段。除紧急缺陷外，不跨阶段混改数据库、权限和页面结构。

状态说明：

- `[ ]` 未开始
- `[-]` 进行中
- `[x]` 已完成并验证

## 总体原则

- 保留现有有效数据，删除模型或字段前必须先做数据迁移和数据库备份。
- 学校身份、课题组身份和系统权限分别表达，不能再用一个角色字段承担全部含义。
- 前端重构不改变现有 API 行为；API 调整必须先补测试，再同步前端。
- 页面列表统一支持搜索、分页、返回后保留页码和筛选条件。
- 文件统一支持上传进度、大小显示、权限校验、预览和下载。
- 每个阶段单独提交，避免形成无法审查的大型提交。

## 阶段一：整理身份、权限和毕业成员逻辑

### 目标

建立稳定、易解释的账号体系，明确“这个人在学校中的身份”“在课题组中的状态”和“能在系统中做什么”是三件不同的事。

### 任务

- [x] 盘点 `accounts`、`members`、`students` 中重复的身份字段、角色枚举和判断逻辑。
- [x] 定义学校身份：硕博导师、博士后、博士生、硕士生、本科生、其他成员。
- [x] 定义成员状态：在组、已毕业/离组；成员状态不直接等同于账号是否可登录。
- [x] 定义系统权限：系统管理员、网站编辑、资料管理员、仪器管理员；普通成员不作为系统权限。
- [x] 明确 `User.is_active` 只控制账号能否登录，不用于表达毕业或离组。
- [x] 建立毕业处理规则：成员身份转为“已毕业”，保留历史成果和学生档案，不自动删除账号。
- [x] 毕业或离组后默认保留账号和查看权限；是否允许登录由账号启用状态单独控制。
- [x] 保证一个账号最多关联一个学生档案，硕博导师和博士后不自动生成学生档案。
- [x] 保留学生与多个导师的关联；毕业后档案进入历史状态，但仍可按权限查看。
- [x] 统一后台权限判断，避免只依赖前端隐藏按钮。
- [x] 编写数据迁移，归一化旧角色、成员类型和毕业状态。
- [x] 更新成员管理页，让身份、成员状态、系统权限分区显示和编辑。

### 权限基线

| 操作 | 管理员 | 内容/资料管理员 | 普通成员 | 文件本人 |
| --- | --- | --- | --- | --- |
| 管理账号和系统权限 | 允许 | 禁止 | 禁止 | 禁止 |
| 编辑门户内容 | 允许 | 内容管理员允许 | 禁止 | 禁止 |
| 查看组内资料 | 允许 | 允许 | 允许 | 允许 |
| 编辑或删除内部资料 | 允许 | 资料管理员允许 | 禁止 | 无资料管理权限时禁止 |
| 查看学生档案及归档资料 | 允许 | 已审核组内成员均允许 | 已审核组内成员均允许 | 允许 |
| 删除学生档案或文件 | 允许 | 禁止 | 禁止 | 仅本人文件允许 |

### 验收标准

- [x] 页面能够明确区分学校身份、成员状态和系统权限。
- [x] 毕业不会造成成员资料、成果署名和学生档案丢失。
- [x] 停用账号后无法登录，但公开团队成员资料可按排序继续展示。
- [x] 普通成员无法通过直接调用 API 越权管理账号或删除他人文件。
- [x] 旧数据迁移后不存在一个账号关联多个学生档案的情况。
